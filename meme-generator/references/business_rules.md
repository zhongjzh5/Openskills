# 表情包生成器业务规则

## 1. 核心业务逻辑

### 1.1 表情包匹配算法
```
输入分析 → 情感识别 → 话题提取 → 模板匹配 → 文字生成 → 图片合成
```

### 1.2 优先级规则
1. **用户偏好优先**：用户设置的风格和喜好具有最高优先级
2. **上下文相关**：当前对话情景比历史对话更重要
3. **流行度加权**：热门模板和趋势话题获得额外权重
4. **安全过滤**：敏感内容自动过滤，需要用户确认

### 1.3 质量评分标准
```yaml
匹配分数计算公式:
  基础分: 0.5
  + 风格匹配: +0.2 (如果用户偏好与模板风格一致)
  + 偏好模板: +0.3 (如果是用户收藏的模板)
  + 话题相关: +0.1 (如果模板标签与话题匹配)
  + 情感匹配: +0.1 (如果模板适合当前情感)
  总分范围: 0.0 - 1.0
```

## 2. 内容生成规则

### 2.1 文字生成规则
- **长度限制**：每段文字不超过20个字符
- **情感适配**：根据检测到的情感选择合适的表达
- **话题聚焦**：文字内容应与对话话题相关
- **幽默分级**：根据用户设置的幽默等级调整搞笑程度

### 2.2 模板选择规则
1. **风格匹配度**：优先选择与用户偏好风格一致的模板
2. **话题相关性**：标签与当前话题匹配的模板优先
3. **使用频率**：用户常用模板获得额外权重
4. **多样性控制**：避免连续使用同一模板

### 2.3 图片处理规则
- **尺寸标准**：
  - Small: 300×300px
  - Medium: 600×400px (默认)
  - Large: 800×600px
- **格式要求**：输出PNG格式，支持透明背景
- **质量设置**：根据输出用途调整压缩质量

## 3. 用户偏好管理

### 3.1 偏好收集规则
- **显式设置**：用户直接设置的偏好具有最高优先级
- **隐式学习**：从用户选择中学习，但不覆盖显式设置
- **频率权重**：经常选择的选项获得更高权重
- **时间衰减**：长期未使用的偏好逐渐降低权重

### 3.2 偏好应用规则
```yaml
应用顺序:
  1. 用户当前设置的偏好
  2. 用户历史选择的模式
  3. 系统默认设置
  4. 通用最佳实践
```

### 3.3 个性化学习规则
- **正面反馈**：用户选择的表情包会强化相关特征
- **负面反馈**：用户跳过的表情包会弱化相关特征
- **话题关联**：特定话题下的偏好会关联到类似话题
- **时间上下文**：偏好可能随时间（如节假日）变化

## 4. 外部API使用规则

### 4.1 API选择规则
1. **免费优先**：优先使用免费API，减少用户成本
2. **质量优先**：在同等条件下选择质量更高的API
3. **速度优先**：响应时间快的API获得优先权
4. **可靠性优先**：选择稳定性高的API服务

### 4.2 限制管理规则
- **速率限制**：严格遵守各API的请求频率限制
- **配额管理**：监控API使用量，避免超额
- **故障转移**：当主API失败时自动切换到备用API
- **成本控制**：避免产生意外费用，需要用户确认

### 4.3 数据缓存规则
```yaml
缓存策略:
  热门模板: 缓存24小时
  搜索结果: 缓存1小时
  用户数据: 永久缓存（除非用户删除）
  API响应: 根据HTTP头决定缓存时间
```

## 5. 安全与合规规则

### 5.1 内容过滤规则
- **敏感词过滤**：自动过滤政治、暴力、色情等敏感内容
- **版权检查**：避免使用受版权保护的图片
- **个人隐私**：不处理包含个人隐私信息的图片
- **文化敏感**：尊重不同文化背景的敏感性

### 5.2 用户确认规则
需要用户确认的操作：
1. 下载超过10张图片
2. 使用付费API服务
3. 生成可能敏感的内容
4. 修改系统配置
5. 删除用户数据

### 5.3 数据隐私规则
- **本地存储**：用户偏好和生成历史存储在本地
- **不收集**：不收集用户对话内容
- **可删除**：用户可以随时删除所有本地数据
- **透明说明**：清楚说明数据如何被使用

## 6. 性能优化规则

### 6.1 响应时间目标
- **本地生成**：< 2秒
- **API搜索**：< 5秒
- **批量处理**：< 30秒（每10个）

### 6.2 资源使用限制
- **内存使用**：单次生成不超过100MB
- **磁盘空间**：缓存不超过500MB
- **网络流量**：单次会话不超过50MB
- **CPU时间**：单次生成不超过10秒

### 6.3 并发处理规则
- **最大并发**：最多同时处理3个请求
- **队列管理**：超过限制的请求进入队列等待
- **超时处理**：超过30秒的请求自动取消
- **错误恢复**：失败的任务可以重试（最多3次）

## 7. 错误处理规则

### 7.1 错误分类
```yaml
错误级别:
  Critical: 系统无法运行
  Error: 单次请求失败
  Warning: 功能降级但仍可用
  Info: 正常但需要注意的情况
```

### 7.2 恢复策略
1. **重试机制**：网络错误自动重试（最多3次）
2. **降级方案**：高级功能失败时使用基础功能
3. **备用方案**：主要方法失败时使用备用方法
4. **用户通知**：重要错误通知用户并提供解决方案

### 7.3 日志记录规则
- **详细程度**：根据错误级别记录不同详细程度的日志
- **隐私保护**：日志中不包含用户敏感信息
- **保留时间**：日志保留30天，错误日志保留90天
- **分析使用**：日志用于系统优化和问题诊断

## 8. 更新与维护规则

### 8.1 模板更新规则
- **定期检查**：每周检查模板库更新
- **用户贡献**：支持用户提交新模板
- **质量审核**：新模板需要经过审核
- **版本管理**：模板更新时保持向后兼容

### 8.2 API更新规则
- **兼容性检查**：API更新时检查接口兼容性
- **逐步迁移**：新API逐步替换旧API
- **回滚计划**：更新失败时可以快速回滚
- **文档更新**：API变更时同步更新文档

### 8.3 系统更新规则
1. **版本规划**：每月发布小版本，每季度发布大版本
2. **测试要求**：更新前必须通过完整测试套件
3. **用户通知**：重大更新提前通知用户
4. **数据迁移**：更新时确保用户数据安全迁移
